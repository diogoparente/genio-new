# Génio — Product Requirements Document

**Status:** Draft
**Date:** 2026-05-15
**Brand:** génio (Portuguese — noun: exceptional intellectual or creative power)

---

## Problem Statement

Aspiring founders and indie developers waste months building products nobody wants. They guess at what to build based on intuition, anecdote, or generic "startup idea" lists. These lists don't tell you whether anyone is actually searching for the solution, discussing the problem in communities, or whether competitors already dominate the space.

Users need a tool that generates specific, actionable micro-SaaS ideas where each idea is backed by verifiable evidence of audience demand — search volume, community discussion signals, and competitive landscape data — so they can build with conviction instead of hope.

---

## Solution

Génio is a web application that generates micro-SaaS business ideas validated by real market signals. A user picks an optional niche, triggers a generation run, and receives a batch of structured ideas. Each idea includes a confidence score, the market signals that support it, competitor analysis, suggested tech stack, pricing guidance, estimated TAM, acquisition channels, and an MVP feature set scoped to 4–8 weeks of solo development.

Every idea also includes a **one-click bootstrap prompt** — a structured AI coding prompt that the user copies into Claude Code, Cursor, or similar tools to scaffold the entire project immediately.

## User Stories

### Authentication

1. As a new user, I want to create an account with email and password, so that I can access the idea generator.
2. As a returning user, I want to sign in with email and password, so that I can access my saved ideas and generation history.
3. As an authenticated user, I want my session persisted across browser restarts, so that I don't need to log in every time.
4. As a user, I want protected routes to redirect me to the login page when I'm not authenticated, so that I can't accidentally access dashboard features without logging in.

### Landing Page

5. As a visitor, I want to see a landing page that explains what the product does, so that I can decide whether to sign up.
6. As a visitor, I want to see feature highlights (market signals, competitive analysis, confidence scoring, bootstrap prompts), so that I understand the full value proposition.
7. As a visitor, I want to see a step-by-step "how it works" section, so that I know what to expect after signing up.
8. As a visitor, I want to see pricing information, so that I understand the cost before committing.
9. As a visitor, I want to see an FAQ section, so that common questions are answered without contacting support.
10. As a visitor, I want to see a final call-to-action section, so that I'm prompted to sign up after learning about the product.

### Internationalization

11. As a non-English speaker, I want to switch the UI language, so that I can use the app in my preferred language.
12. As a user, I want my language preference persisted across sessions via a cookie, so that I don't need to re-select it every time I visit.
13. As a user, I want the language switcher accessible from every page (landing and dashboard), so that I can change language at any time.

### Generation

14. As a user, I want to trigger an idea generation run, so that I receive a batch of validated micro-SaaS ideas.
15. As a user, I want to optionally specify a niche before generating, so that the ideas are relevant to markets I care about.
16. As a user, I want to select how many ideas to generate (via a slider, 3–10), so that I control the batch size.
17. As a user, I want to click suggested niche chips for quick selection, so that I can explore popular categories without typing.
18. As a user, I want to generate ideas without a niche (broad exploration), so that I can discover unexpected opportunities.
19. As a user, I want to see a loading state with an estimated time while ideas are being generated, so that I know the system is working.
20. As a user, I want the generation to complete within a reasonable time (15–45 seconds), so that I'm not waiting indefinitely.

### Idea Browsing

21. As a user, I want to browse all my generated ideas in a paginated grid, so that I can review them at my own pace.
22. As a user, I want to filter ideas by monetization model (subscription, one-time, usage-based, hybrid), so that I can focus on ideas with a business model I prefer.
23. As a user, I want to filter ideas by minimum confidence score, so that I only see high-viability ideas.
24. As a user, I want to filter to show only saved ideas, so that I can focus on my shortlist.
25. As a user, I want each idea card to show the name, tagline, confidence score, monetization model, and signal badges, so that I can quickly assess ideas without opening each one.
26. As a user, I want to see trend direction badges (up/flat/down) per signal source on each idea card, so that I can gauge market momentum at a glance.

### Idea Detail

27. As a user, I want to view the full details of a single idea, so that I can deeply evaluate its viability.
28. As a user, I want to see the market signals that support an idea (keyword, volume, trend, mentions, sentiment summary), so that I can verify the evidence behind it.
29. As a user, I want to see existing competitors with their name, URL, description, and competitive strength, so that I understand the competitive landscape.
30. As a user, I want to see a suggested tech stack, so that I know what tools to use if I build it.
31. As a user, I want to see an estimated TAM description, so that I understand the market size.
32. As a user, I want to see pricing suggestions (plan name, price, features), so that I have a starting point for monetization.
33. As a user, I want to see suggested customer acquisition channels, so that I know how to find early users.
34. As a user, I want to see an MVP feature set scoped to 4–8 weeks, so that I know exactly what to build first.
35. As a user, I want to save or unsave an idea for later reference, so that I can curate a shortlist of promising ideas.
36. As a user, I want to delete an idea I'm not interested in, so that my idea list stays manageable.

### Bootstrap Prompt

37. As a user, I want to copy a structured AI coding prompt that includes the idea's name, description, target audience, tech stack, and MVP features, so that I can paste it into Claude Code or Cursor to scaffold the project immediately.
38. As a user, I want visual confirmation when the prompt has been copied to clipboard, so that I know the copy was successful.

### Generation History

39. As a user, I want to see a list of all my past generation runs, so that I can revisit previous batches.
40. As a user, I want each history entry to show the niche (if any), batch size, status, and date, so that I can identify the run I'm looking for.
41. As a user, I want to click into a generation run to see all ideas from that batch, so that I can review them together.

### Data Sources & Health

42. As a developer, I want a health check endpoint that reports the status of all data source adapters, so that I can monitor the system.
43. As a user, I want generation to succeed even if some data sources are unavailable (graceful degradation), so that a single source outage doesn't block the experience.

## Implementation Decisions

### Architecture

- **Single Next.js App Router application** — Server components for data fetching, client components for interactivity (forms, filters, modals). API routes handle auth, generation, and CRUD operations.
- **Synchronous generation** — The generation API route runs the full pipeline (fetch signals → LLM synthesize → store) and returns results. No background jobs, no polling. This keeps the system self-contained with no Redis or queue dependency.
- **Graceful degradation for data sources** — Each source is queried independently via `Promise.allSettled`. A single source failure does not block the generation run; the LLM synthesizes from whatever signals were successfully retrieved.

### Tech Stack

- **Next.js 16** (App Router) with TypeScript
- **better-auth** for authentication — email/password enabled, session management via cookies, `nextCookies()` plugin for Next.js integration
- **Drizzle ORM + Neon Postgres** — schema defined with Drizzle's pg-core, migrations managed via `drizzle-kit push`
- **LLM Provider** — Configurable via environment variables (`LLM_API_KEY`, `LLM_BASE_URL`, `LLM_MODEL`). Defaults to DeepSeek API. OpenAI SDK included as an alternative. The provider is a stateless function, not a class, that calls any OpenAI-compatible chat completions endpoint.
- **Tailwind CSS v4** with a custom neumorphic design system
- **i18next + react-i18next** for internationalization, cookie-based language persistence
- **pnpm** as the package manager

### Design System

- **Neumorphic aesthetic** — Soft extruded shadows, inset wells, rounded corners (24–32px), muted color palette on a light gray-blue background (`#e0e5ec`).
- **Fonts** — Plus Jakarta Sans for display/headings, DM Sans for body text.
- **Accent color** — Warm coral (`#ff9a75`) for primary actions and highlights. Secondary accent: teal (`#38b2ac`).
- **CSS custom properties** — All design tokens defined as CSS variables (`--color-neu-bg`, `--color-neu-accent`, etc.) in `globals.css`.
- **Shadow utilities** — Dedicated utility classes for neu shadows: raised (`shadow-neu`, `shadow-neu-sm`), hover (`shadow-neu-hover`), inset (`shadow-neu-inset`, `shadow-neu-inset-deep`, `shadow-neu-inset-sm`), accent-colored (`shadow-neu-accent`, `shadow-neu-accent-hover`, `shadow-neu-accent-inset`).
- **Reduced motion support** — Animations disabled when `prefers-reduced-motion: reduce`.

### Internationalization

- **7 supported languages** — English, Portugues, Espanol, Francais, Deutsch, Chinese, Japanese.
- **Translation files** — JSON key-value pairs in `src/locales/{code}/translation.json`.
- **Server-side** — `getServerT()` creates an i18next instance, detects language from the `lang` cookie, and returns a bound `t()` function.
- **Client-side** — `I18nProvider` wraps the app with `I18nextProvider`, hydrates from the cookie on mount.
- **Language switcher** — Dropdown component accessible from the landing page header and the dashboard navbar. Sets the `lang` cookie with a 1-year max-age and calls `router.refresh()` to re-render server components with the new locale.
- **All user-facing strings** are extracted to translation files — no hardcoded English text in components.

### Data Model

- **Users** — Managed by better-auth (user, session, account, verification tables).
- **Idea Generations** — One row per generation run. Fields: id, userId, niche (nullable), batchSize, status (pending/running/complete/failed), confidence (average across ideas), errorMessage (nullable), createdAt.
- **Ideas** — Individual generated ideas. Fields: id, generationId, name, tagline, description, targetAudience, monetizationModel (subscription/one_time/usage_based/hybrid), confidenceScore (0–100), isSaved (0/1), notes (nullable), createdAt. Indexed on generationId.
- **Idea Signals** — Market evidence per idea. Fields: id, ideaId, source (google_trends/reddit/hackernews/producthunt), keyword, volumeEstimate, trendDirection (up/flat/down), mentionCount, sentimentSummary, rawData (jsonb), createdAt. Indexed on ideaId.
- **Idea Competitors** — Existing players. Fields: id, ideaId, name, url, description, strength (strong/moderate/weak), createdAt. Indexed on ideaId.
- **Idea Details** — LLM-synthesized enrichment. Fields: id, ideaId, suggestedTechStack (jsonb array), estimatedTAM, acquisitionChannels (jsonb array), pricingSuggestions (jsonb array of objects), mvpFeatureSet (jsonb array), createdAt. Unique index on ideaId (1:1 relationship).

### Generation Pipeline

1. User sends POST to `/api/generations` with optional `niche` and `batchSize` (default 7)
2. A generation row is created with status `running`
3. All four data sources (Google Trends, Reddit, Hacker News, Product Hunt) are queried in parallel with `Promise.allSettled`
4. Each source normalizes output to a common `NormalizedSignal` interface: `{ keyword, volume, trend, mentionCount, sentimentSummary, rawData }`
5. Normalized signals are fed to the LLM via a structured system prompt and a data-rich user prompt
6. The LLM returns JSON validated against a Zod schema — invalid entries are caught at parse time
7. The full batch is written to the database: ideas, signals, competitors, and details
8. The generation is marked `complete` with an average confidence score
9. If anything fails, the generation is marked `failed` with the error message stored

### API Routes

| Method   | Path                      | Purpose                                        | Auth Required |
| -------- | ------------------------- | ---------------------------------------------- | ------------- |
| POST     | `/api/auth/sign-up/email` | Create account                                 | No            |
| POST     | `/api/auth/sign-in/email` | Sign in                                        | No            |
| GET/POST | `/api/auth/[...all]`      | better-auth catch-all handler                  | Varies        |
| POST     | `/api/generations`        | Trigger new idea generation                    | Yes           |
| GET      | `/api/generations`        | List past generation runs for user             | Yes           |
| GET      | `/api/generations/[id]`   | Get one generation with its ideas              | Yes           |
| GET      | `/api/ideas`              | Browse ideas (paginated, filterable)           | Yes           |
| GET      | `/api/ideas/[id]`         | Single idea with signals, competitors, details | Yes           |
| PATCH    | `/api/ideas/[id]`         | Save/unsave an idea, update notes              | Yes           |
| DELETE   | `/api/ideas/[id]`         | Remove an idea                                 | Yes           |
| GET      | `/api/sources`            | Data source health check                       | No            |

### Route Protection

- Only better-auth authenticated requests can redirect to /dashboard

### Shared Libraries (Deep Modules)

- **`src/lib/ideas/reader.ts`** — Encapsulates all idea queries: `listIdeas()` with filtering/pagination, `getIdeaWithRelations()`, `verifyIdeaOwnership()`, `fetchSignalsForIdeas()`. Single interface that pages and API routes consume.
- **`src/lib/generations/reader.ts`** — Encapsulates generation queries: `listGenerations()`, `getGenerationWithIdeas()`. Single interface consumed by generation history pages.
- **`src/lib/i18n/`** — Server-side translation (`server.ts`), client-side provider (`I18nProvider.tsx`), language constants (`languages.ts`), and re-export barrel (`index.ts`). Single module for all i18n concerns.

### Data Source Adapters

Each adapter implements the `SignalSource` interface (`name`, `fetchSignals()`, `healthCheck()`). The registry module (`registry.ts`) orchestrates parallel fetching and health checks. Sources:

- **Google Trends** — Fetches trend data for seed keywords, parses timeline values, computes trend direction from recent vs. older averages
- **Reddit** — Searches Reddit's public JSON API for pain-point language patterns, extracts engagement metrics and subreddit attribution
- **Hacker News** — Uses Algolia's HN search API to find discussions and launches, scores by points and comments
- **Product Hunt** — Queries the GraphQL API for trending products by topic, normalizes votes and comments into signals

### Frontend Components

- **`GenerationForm`** — Niche text input with suggested niche chips, batch size slider (3–10), submit button with loading state, error display. Client component.
- **`GenerationStatus`** — Spinner with status text shown during synchronous generation (15–45s). Client component.
- **`IdeaCard`** — Compact card: name, tagline, confidence score badge (color-coded: green ≥80, yellow ≥60, red <60), monetization model label, signal source badges with trend arrows, saved indicator. Links to idea detail. Client component.
- **`IdeaDetail`** — Full idea view with sections: description + target audience, market signals (with keyword, volume, mentions, sentiment), competitors grid (with name, URL, strength), tech stack chips, estimated TAM, pricing plan cards, acquisition channels list, MVP feature set list. Includes save/unsave toggle button. Client component.
- **`SignalBadge`** — Inline pill showing source abbreviation (Google/Reddit/HN/PH) and trend arrow (↑/→/↓) with color coding (green/yellow/red).
- **`IdeaFilters`** — Dropdown filters for monetization model, confidence threshold, and saved status. Updates URL search params. Client component.
- **`BootstrapPrompt`** — Structured AI prompt builder and copy-to-clipboard. Displays tech stack chips, the generated prompt in a `<pre>` block, and a copy button with "Copied!" confirmation. Client component.
- **`Navbar`** — Sticky top nav with brand link (→ `/dashboard`), navigation links (Generator, Ideas, History), and language switcher. Client component.
- **`LanguageSwitcher`** — Dropdown with 7 language options, sets cookie, changes i18n language, refreshes router. Client component.
- **`AuthForm`** — Email/password form with login/signup modes, validation, loading state, error display. Client component.

### Pages

- **`/`** — Landing page with: sticky header (brand + language switcher + Sign In / Get Started buttons), hero section, features grid (4 cards), how-it-works (3 steps), stats bar, social proof section, pricing card (Beta/$0), FAQ accordion, final CTA, footer. Server component using `getServerT()` for translations.
- **`/login`** — Auth layout + AuthForm in login mode + link to signup. Server component shell with client form.
- **`/signup`** — Auth layout + AuthForm in signup mode + link to login. Server component shell with client form.
- **`/dashboard`** — Hub page: GenerationForm section at top, recent ideas grid below (up to 6 from last 3 generations). Server component that fetches ideas and signals.
- **`/dashboard/ideas`** — All ideas browse page: IdeaFilters bar + paginated IdeaCard grid. Server component with search params for filtering.
- **`/dashboard/ideas/[id]`** — Idea detail page: back link + IdeaDetail component with full data (idea, signals, competitors, details) + BootstrapPrompt section. Server component that verifies ownership.
- **`/dashboard/generations`** — Generation history page: list of past runs with niche, batch size, status badge, and date. Server component.
- **`/dashboard/generations/[id]`** — Generation detail page: header (niche, date, status) + grid of IdeaCards for that batch. Server component that verifies ownership.

## Testing Decisions

### What Makes a Good Test

- Test external behavior, not implementation details
- API routes should be tested end-to-end against a real database (test Neon instance or local Postgres)
- Use the same Zod schemas used in production for response validation
- Test error paths: unauthorized access, missing resources, LLM failures, source degradation
- Client components should be tested for correct rendering of all states (loading, empty, error, success) using React Testing Library

### Modules to Test

- **Generation Pipeline** (`src/lib/generation/orchestrator.ts`) — End-to-end test: mock data sources, mock LLM, verify correct DB writes and status transitions. Test failure mode: LLM throws → generation marked failed.
- **API Routes** — Test each route for auth gating, correct response shapes, filtering/pagination behavior.
- **Idea Reader** (`src/lib/ideas/reader.ts`) — Test filtering logic, pagination offset math, ownership verification.
- **Generation Reader** (`src/lib/generations/reader.ts`) — Test listing and detail retrieval with ownership scoping.
- **Source Registry** (`src/lib/sources/registry.ts`) — Test that `Promise.allSettled` correctly handles mixed success/failure scenarios.
- **Synthesis Parser** (`src/lib/synthesis/parser.ts`) — Test that the Zod schema accepts valid LLM output and rejects malformed JSON (missing fields, wrong types, out-of-range confidence scores).

### Prior Art

- API route tests follow the standard Next.js pattern: construct a `Request`, call the handler, assert on `NextResponse.json()` output
- Drizzle queries tested against a real database using a test schema migration
- Component tests follow React Testing Library conventions for accessible queries

## Out of Scope

- Automated recurring generation (manual trigger only)
- Team or organization accounts
- Export to CSV or PDF
- Payment or monetization of the génio tool itself (free during beta)
- Passkeys, MFA, OAuth social login, password reset flows (email/password is the MVP)
- Stripe integration or paid plans
- Email verification
- Rate limiting
- Roles and permissions (admin/user)
- SSE or WebSocket streaming for generation progress (synchronous API + loading spinner is sufficient for the 15–45s generation time)
- Real-time collaboration
- Mobile native apps (web-only, responsive)
- Dark mode (neumorphic light theme only)
- A/B testing framework
- Analytics or telemetry

## Further Notes

### Template-Based Rebuild

The current implementation was built incrementally. A rebuild from scratch should use the Next.js template at `nextjs/` as the foundation. This template is a better-auth demo app that already wires up the full auth stack (email/password, session management, Next.js integration). The rebuild process should:

1. Start from the `nextjs/` template scaffold
2. Port the design system (fonts, neu shadows, color tokens) to the new `globals.css`
3. Port the Drizzle schema with all tables (user, session, account, verification, idea_generations, ideas, idea_signals, idea_competitors, idea_details)
4. Port the shared library modules deep modules — idea reader, generation reader, i18n system, LLM provider
5. Rebuild API routes using the template's auth patterns
6. Port all frontend components with the neumorphic design language
7. Port all translation files and wire up the I18nProvider in the root layout
